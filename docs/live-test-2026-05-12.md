# Live LLM test - 2026-05-12

End-to-end validation of M1-M7 pipeline against a real LLM
(DeepSeek `deepseek-chat`). Single-shot run on the curated
1063-char heavy-AI sample defined in `scripts/verify_e2e.py`.

## Setup

```
provider              : deepseek (OpenAI-compatible endpoint)
model                 : deepseek-chat (default)
sample                : 1063 chars, hand-crafted to fire 13 detector violations
                        across blacklist_words / blacklist_phrases / rhythm /
                        soul_signals buckets
baseline rule score   : 100.0 / 100  (VERY HIGH)
baseline ngram score  : 100.0 / 100  (VERY HIGH)
baseline combined     : 100.0 / 100  (VERY HIGH)
```

## Result matrix

| Strength | Polished rule | Polished combined (= ngram) | Rule drop | Combined drop |
|----------|---------------|-----------------------------|-----------|---------------|
| baseline | 100.0         | 100.0                       | -         | -             |
| LOW      | 16.0          | 99.8                        | **+84.0** | +0.2          |
| MEDIUM   | 14.0          | 99.8                        | **+86.0** | +0.2          |
| HIGH     | 5.0           | 99.6                        | **+95.0** | +0.4          |

## Polished output (first 200 chars per strength)

**LOW** (3-section trimmed rules, 1911-char output):

> Software development is messy. It's not a single path but a tangle
> of decisions about tools, workflows, and how people work together.
> The teams that ship reliably aren't the ones with the most tools-

**MEDIUM** (full 8-section analysis scene, 2825-char output):

> Software development is not a single skill. It is a pile of
> contradictory disciplines--architecture, psychology, operations,
> economics--mashed into one job title. Teams that treat it as a
> single "endeav[our]...

**HIGH** (aggressive rewrite template, 3007-char output):

> March 2024. The traffic curve flatlined. Two months earlier, the

>
> Why do most software projects stall after the first sprint? Not
> because the code is bad, but...

## Observations

1. **Rule layer responds beautifully.** From 100 -> 5 at HIGH strength
   (95-point drop) confirms the M3+M4 rule blacklist + M6 prompt
   pack are aligned: the LLM honoured the `SELF_CHECK` rule names
   pinned to `rules.json` and avoided the flagged constructs.
   M5's `WORDS_BLACKLIST` is doing serious work.

2. **Strength gradient is monotonic.** LOW (+84) < MEDIUM (+86) <
   HIGH (+95). The 11-point gap between MEDIUM and HIGH is the
   `POSTPROCESS_PROMPT_AGGRESSIVE` template doing its job:
   structural rewrite, not just vocabulary swap.

3. **HIGH produces qualitatively different prose.** The HIGH
   output opens with a *scene* ("March 2024. The traffic curve
   flatlined"), uses *concrete dates*, *named subjects*, and a
   *contrarian hinge* ("Not because the code is bad, but..."). This
   is exactly what `SOUL_INJECTION` + `OPENING_DIVERSITY` +
   `ASSERTION_TEMPLATE` were authored to elicit. The standard pass
   (MEDIUM) stays essay-shaped but trades AI vocabulary for
   declarative concrete claims ("contradictory disciplines--
   architecture, psychology, operations, economics--mashed into one
   job title").

4. **N-gram layer barely moved (99.8 -> 99.6).** This is a known
   M2 limitation, not an M3-M7 failure:
   - HC3-en's "human" half is Q&A-style on technical topics.
   - All three polished outputs remain *essay-style* and stay in
     the same lexical distribution as HC3-AI ("teams", "ship",
     "tools" etc. are common to both halves of HC3).
   - The n-gram detector reports domain-style similarity, not
     polish quality. It is doing what we trained it to do.

   This matches the rollback plan in `docs/plan.md` §12: the
   Binoculars-drop gate (M8) is the appropriate measure of polish
   effectiveness against *independent* detectors. The HC3-trained
   n-gram is a lower bound (every AI sample we throw at it scores
   high) but not a useful upper bound for "did we humanize it?".

5. **Output length grows with strength.** LOW 1911 chars, MEDIUM
   2825, HIGH 3007 (vs 1063 input). HIGH especially expands because
   the aggressive template asks for soul + concrete + contrarian
   structure, which is intrinsically more content. Acceptable.

## What this validates

- M1 (scaffold + protocol routing): humanize-core saw the plugin,
  used its detector, replacements, and writer hook.
- M2 (n-gram engine): scored both before and after; n-gram drop is
  small but the *measurement* path works.
- M3-M4 (26 detector rules): 13 violations detected on the sample;
  rule score crashed after polish.
- M5 (102 replacements): used by `_deterministic_cleanup` candidate
  inside `humanize_core.postprocess._best_candidate`, contributed
  to MEDIUM picking the deterministic-cleaned candidate when LLM
  output scored higher.
- M6 (prompt pack): 14k-char MEDIUM prompt actually steered the
  LLM (rule drop of 86 points).
- M7 (strength knob): monotonic gradient LOW < MEDIUM < HIGH on
  rule drop, plus qualitatively different prose at HIGH.

## What still needs work (deferred to M8+)

- N-gram detector is not informative for polished text.
  Either retrain on essay/news corpus, or replace with Binoculars
  perplexity (the plan's M8 / our renumbered M9).
- LOW strength needs an A/B comparison against MEDIUM on
  *less-AI* input (where MEDIUM might be heavy-handed). The
  current sample is so saturated that MEDIUM works regardless.
- Combined-score early-publish short-circuit
  (`humanize_core.postprocess`, line 372) checks
  `score_before.total < 25 AND combined_v < 30`. Our polished
  output now satisfies the first condition (rule < 25) but not the
  second (combined ~99). A second polish pass would not short-
  circuit and would burn another LLM call to no benefit.
  Worth filing against humanize-core: "publish threshold should
  use *combined drop from baseline*, not absolute combined".
